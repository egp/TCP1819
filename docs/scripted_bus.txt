TCP1819 Scripted Bus Implementation Plan
=======================================

Goal
----
Add a TCP1819-owned host-test scripted bus that operates at the BBI2C level and can
be reused by any host test that calls TCP1819-style bus functions.

This plan affects TCP1819 directly. It does not require any production Arduino
behavior changes.

Scope
-----
Provide a host-test helper plus host-only shim implementations that allow tests to:
- bind a BBI2C instance to a scripted bus object
- script probe/test, read, and write outcomes
- observe exact bus traffic in chronological order
- simulate short reads, short writes, and empty/failure-style results
- support multiple independent BBI2C instances in one test process

Non-goals
---------
- no device-specific protocol helpers
- no production-mode mocking
- no Arduino build changes beyond keeping host-only code out of Arduino builds
- no requirement to add convenience wrappers for every TCP1819 API on day one

Why this belongs in TCP1819
---------------------------
TCP1819 is the bus-layer dependency. A reusable host-test helper belongs here so that
direct bus-level tests and thin adapters layered above BBI2C can share one consistent
test seam.

The helper must match the real TCP1819 function contracts:
- I2CInit(...) is void
- I2CTest(...) returns an int-like success result
- I2CRead(...) returns a byte count
- I2CWrite(...) returns a byte count

That means the scripted bus must model counts, not just booleans.

Design rules
------------
1. Keep the helper host-only.
   Put all scripted-bus code outside the normal production source path or guard it so
   it is only part of host-test builds.

2. Keep the helper bus-level.
   The helper should know about BBI2C, addresses, bytes, lengths, and return counts.
   It should not know about sensor frames, register maps, displays, clocks, or any
   higher-level protocol semantics.

3. Support both scripting and observation.
   Tests need to inject results and also inspect exactly what traffic occurred.

4. Preserve operation order.
   Some WB tests need to assert read/write/test interleaving, not just totals.

5. Support more than one bus.
   Multiple BBI2C instances must be independently bindable in the same process.

Recommended file layout
-----------------------
Suggested host-test files under TCP1819:

- extras/host_test/TCP1819ScriptedBus.h
- extras/host_test/TCP1819ScriptedBus.cpp
- extras/host_test/TCP1819HostShim.cpp
- extras/host_test/README.txt

The exact directory name can vary, but the intent is:
- clearly host-test only
- easy for downstream host builds to include and link
- absent from Arduino library compilation

Core model
----------
A TCP1819ScriptedBus instance owns:
- queued scripted results
- accumulated observed operations
- default behaviors used when no explicit script item is queued

A host-only binding registry maps:
- BBI2C*  ->  TCP1819ScriptedBus*

The host-only shim functions:
- I2CInit
- I2CTest
- I2CRead
- I2CWrite

will:
- look up the bound TCP1819ScriptedBus for the passed BBI2C*
- record the operation
- consume scripted results or defaults
- return the correct count or value for the real API contract

Recommended public helper API
-----------------------------
The exact names may vary, but Phase 1 should include equivalents of:

Lifecycle / configuration
- reset()
- clearScript()
- clearObservations()
- setDefaultReadCount(int count)
- setDefaultWriteCount(int count)
- setDefaultTestResult(int result)

Binding
- bind(BBI2C& bus, TCP1819ScriptedBus& scriptedBus)
- unbind(BBI2C& bus)
- unbindAll()      // optional but useful in test teardown

Scripting
- queueTestResult(int result)
- queueReadBytes(const uint8_t* data, int returnedCount)
- queueReadVector(const std::vector<uint8_t>& data, int returnedCount)
- queueWriteCount(int returnedCount)

Optional address-sensitive scripting, if wanted immediately:
- queueTestResultForAddress(uint8_t address, int result)
- queueReadForAddress(uint8_t address, bytes..., int returnedCount)
- queueWriteCountForAddress(uint8_t address, int returnedCount)

Observation
- initCallCount()
- testCallCount()
- readCallCount()
- writeCallCount()

- lastInitClockHz()
- lastTestAddress()
- lastReadAddress()
- lastWriteAddress()

- writeHistoryCount()
- writeBytesAt(size_t index)
- readResultBytesAt(size_t index)

Mixed ordered log
- operationCount()
- operationAt(size_t index)

Consumption / status
- hasPendingScript()
- pendingReadCount()
- pendingWriteCount()
- pendingTestCount()
- scriptFullyConsumed()

Operation log shape
-------------------
Provide one chronological operation log with entries such as:

enum class OpKind {
  Init,
  Test,
  Read,
  Write
};

struct Operation {
  OpKind kind;
  BBI2C* bus;
  uint8_t address;

  uint32_t initClockHz;          // valid for Init
  int requestedLength;           // valid for Read/Write
  int returnedCountOrResult;     // valid for Test/Read/Write

  std::vector<uint8_t> bytes;    // written bytes for Write;
                                 // returned bytes for Read
};

This log is the key feature that enables WB tests to assert exact ordering.

Required semantics
------------------
1. I2CInit
   - record an Init operation
   - store the passed clock for observation
   - no failure path is needed in Phase 1 because the real API is void

2. I2CTest
   - record a Test operation with bus and address
   - return the next queued test result, or the configured default
   - allow explicit simulation of both success and failure

3. I2CWrite
   - record a Write operation with exact address, requested length, and written bytes
   - return the next queued write count, or the configured default
   - defaults should usually be "all bytes accepted" unless configured otherwise
   - support short writes by allowing any returned count from 0..requestedLength

4. I2CRead
   - record a Read operation with exact address and requested length
   - consume the next queued read item, if any
   - copy returnedCount bytes into the caller buffer
   - return returnedCount
   - support short reads by allowing returnedCount < requested length
   - if no read is queued, use the configured default count behavior

5. Default behavior
   Suggested defaults:
   - test default: success
   - write default: requested length
   - read default: 0 bytes returned

   These defaults make missing script items visible while still being deterministic.

6. Script persistence
   Observations must remain queryable after script items are consumed.

7. Isolation
   Traffic on one bound BBI2C instance must not affect another.

Address-sensitive scripting
---------------------------
Address-sensitive queue items are optional in the initial slice, but recommended.
They help catch wrong-address bugs earlier.

If implemented in Phase 1, document deterministic mismatch behavior:
- either fail the test immediately with a clear assertion
- or record the mismatch and return a configurable failure result

Do not silently accept an address mismatch.

Recommended helper behavior choices
-----------------------------------
1. Prefer vectors over raw pointers internally.
2. Copy scripted read data into owned storage when queued.
3. Record exact written bytes even when the scripted return count is short.
4. Keep script and observation clearing separate.
5. Make teardown easy:
   reset() should leave the object in a fresh state.

Phase 1 deliverables
--------------------
1. TCP1819ScriptedBus class with:
   - queued test/read/write scripting
   - ordered mixed operation log
   - count and "last operation" observation helpers
   - reset / clear APIs

2. Host-only binding shim for:
   - I2CInit
   - I2CTest
   - I2CRead
   - I2CWrite

3. TCP1819-owned host tests for:
   - init logging
   - test result scripting
   - write count scripting and payload capture
   - read count scripting and returned byte capture
   - short write behavior
   - short read behavior
   - default behaviors when queues are empty
   - ordered interleaving of mixed operations
   - multiple bound bus isolation

Phase 2 candidates
------------------
Only add these if a real consumer needs them:

- host-only shim support for I2CReadRegister
- host-only shim support for I2CScan
- convenience helpers for exact expected sequences
- stricter address-aware scripting and mismatch diagnostics
- optional helpers to simplify common register-pointer write/read idioms

Acceptance criteria
-------------------
The change is complete when TCP1819 provides:

1. A host-test scripted bus owned by TCP1819.
2. Host-only shim implementations for the core BBI2C-level free functions:
   - I2CInit
   - I2CTest
   - I2CRead
   - I2CWrite
3. Scripted support for:
   - test/probe outcomes
   - read byte buffers plus returned counts
   - write returned counts
4. Exact observation of:
   - addresses
   - requested lengths
   - returned counts/results
   - written bytes
   - read-returned bytes
   - mixed chronological operation order
5. Correct isolation between multiple simultaneously bound BBI2C instances.
6. No required production-library API changes.
7. No device-specific logic added to TCP1819.

Suggested implementation order
------------------------------
Step 1
Add failing TCP1819 host tests that define:
- bind/unbind behavior
- init/test/read/write routing
- short count semantics
- ordered operation logging
- multi-bus isolation

Step 2
Implement TCP1819ScriptedBus storage and scripting queues.

Step 3
Implement the host-only shim functions and binding registry.

Step 4
Make the mixed operation log and helper accessors green.

Step 5
Add a short README.txt showing:
- how to bind a BBI2C
- how to queue a read
- how to queue a write count
- how to inspect ordered traffic

Step 6
Defer any optional APIs such as I2CReadRegister or I2CScan until an actual host test
needs them.

Notes for future users
----------------------
This helper is intentionally low-level. Tests built on top of it should decide locally
how much abstraction they want:
- direct BBI2C-level tests can bind the bus and call TCP1819 functions directly
- higher-level host tests can keep thin local adapters if that improves readability

The TCP1819-owned asset is the reusable bus seam itself.
